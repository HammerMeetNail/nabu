## 1. Executive Summary

Nabu is a well-engineered project. The Go backend is clean, minimally-dependent (3 direct deps), and the security fundamentals are genuinely good. The two prior security reviews drove real fixes, and I **verified the previously-open critical/high/medium findings are now closed** (IDOR, OIDC signature/nonce, invite-deletion ownership, stats household scoping, session-token-in-JSON, Go stdlib CVEs).

The headline risks today are no longer classic web vulnerabilities — they are **two structural/operational issues**:

1. **iOS has drifted from the PWA, and the tracking that should catch it has broken down.** The PWA shipped **214 commits in the last 90 days** vs **21 for iOS**. The iOS app is actually *substantially built* (a full 5-tab SwiftUI app — see §3.2) — but the parity matrix marks ~50 of ~55 features "iOS pending," so it badly **understates** what exists while **failing to capture** what the PWA has since changed. Net result: nobody can trust the matrix in either direction. A concrete live divergence: the PWA removed Day/Week from the Activity tab *today* (`9aeedd5`), but iOS still ships all three Activity modes. And **native push is non-functional** — iOS defines APNs request models but has no registration code, and the backend never routes `/api/mobile/apns/*`, despite the matrix marking APNs "Done." Given iOS is in active development toward a release, this is the most important area to address. (Full analysis and root cause in §3.)

2. **Rate limiting / audit IP attribution silently depends on `TRUSTED_PROXY_CIDRS`, which ships empty.** Behind Cloudflare Tunnel, if this isn't set in production, every request appears to originate from one source IP — collapsing the auth rate limiter into a single shared bucket (global false lockouts *or* ineffective brute-force protection) and rendering audit-log IPs meaningless. **This needs a one-line config verification.**

Everything else below is hardening, polish, and process.

## 2. Security Review

### 2.1 Strengths (keep doing these)

- **Parameterized SQL throughout** via `pgx/v5`; no string-built queries.
- **bcrypt** password hashing; **`crypto/rand`** for all tokens (session 32B, verify/magic/reset 24B), **SHA-256-hashed at rest** in the token/session stores.
- **CSRF**: double-submit cookie + `subtle.ConstantTimeCompare` (`internal/middleware/csrf.go`).
- **Session hygiene**: 24h idle timeout + 30d absolute, session rotation + global invalidation on password change/reset (`auth/service.go:340,387`).
- **Strong headers + CSP** (`internal/middleware/security.go`): `default-src 'self'`, `object-src 'none'`, `frame-ancestors 'none'`, HSTS when TLS.
- **OIDC done correctly**: RS256 signature via JWKS, `iss`/`aud`/`exp`/`nonce` all verified, state checked with constant-time compare (`auth/oidc.go`, `handlers/auth.go:265`).
- **Authorization is consistent**: every resource handler scopes by `*user.HouseholdID`, and services re-check ownership (defense in depth — e.g. `handlers/log.go:138`).
- **Request body cap** (`http.MaxBytesReader`, 1 MB) in `handlers/json.go`.
- **Supply chain**: Gitleaks + Trivy + Cosign keyless signing in CI; non-root container; pinned Go `1.25.11` (current stdlib CVE fixes).

## 3. PWA ↔ iOS Feature Drift  *(primary concern)*

**Conclusion: yes, iOS has drifted — but not in the way the project's own docs suggest.** The iOS app is *more* complete than the parity matrix claims, while the matrix is *less* current than the PWA. The tracking system that was supposed to prevent this exists but doesn't actually verify anything. iOS is in active development (not shipped), so some lag is expected; the real problem is that the lag is **invisible** to the team's tooling.

### 3.1 What iOS actually is today (the matrix is wrong in the optimistic direction)

The native app on `main` is a **functioning 5-tab SwiftUI client**, not a skeleton — *(confirmed by build: the `Nabu` app target compiles clean under Xcode 26.5 for the iOS 26.2 simulator, `** BUILD SUCCEEDED **`)*. `ContentView.swift` wires up a `TabView` with **Stats / Activity / Home / Schedule / Settings** — the same tab set and order as the PWA — plus:

- **Auth & onboarding**: `LoginView`, `RegisterView`, `MagicLinkView`, `OnboardingView`, `GoogleOAuthCoordinator`, session persistence (`auth.loadSession()`), and a CSRF pre-flight (`GET /api/me`) to obtain the `nabu_csrf` cookie the native client can't get from an HTML page load.
- **Data layer**: `APIClient`, `CookieStore`, `CSRFTokenProvider`, typed `Models`/`RequestModels`, per-domain stores (`Chore`/`Log`/`Schedule`/`Activity`) and `DataLoader`s, plus `foregroundRefresh()` on `willEnterForeground`.
- **Feature views**: `HomeView`/`HomeGrid`/`QuickLogSheet`, `LogSheet`, `ManageChoresView`/`ChoreEditView`, `ScheduleView`, `StatsView`, `HouseholdView`, `NotificationsView`/`NotificationPreferencesView`, `UndoToast`.
- **Tests**: 11 XCTest files (contract/decoding/encoding/per-domain) + `NabuUITests` + a `MockAPIServer`.

Yet the matrix marks **Home grid, Chore CRUD, Stats Overview, Schedule CRUD, In-app notifications — all "iOS pending"** even though `HomeView`, `ChoreEditView`, `StatsView`, `ScheduleView`, and `NotificationsView` all exist and ship. **The matrix is stale, not the app.** This matters because it makes the matrix useless as a planning tool and silently breaks the one CI check that depends on it (§3.4).

> Note: I confirmed the latest iOS code lives on `main`. The `ios-plan-phase1` worktree/branch is *older* (last commit 2026-06-07) than `main`'s iOS state (2026-06-18), and `reconcile-ios` is already fully merged. There is no newer iOS work hiding in a branch.

### 3.2 Confirmed live divergences (PWA changed, iOS didn't)

- **Activity tab structure — verified by reading both sides.** PWA commit `e9a9527`: *"Remove day/week view tabs from Activity section, keep history only."* iOS `App/NavigationModel.swift` defined `enum ActivityViewMode { case history, day, week }`, and `Views/ActivityView.swift` rendered a `PillTabBar` over all three (`HistoryListView` / `DayView` / `WeekView`). **iOS showed two Activity modes the PWA deleted** — the clearest, current, user-visible drift. **→ Addressed:** iOS Activity is now history-only (DayView/WeekView and supporting state removed). Note the PWA still retains *unrouted* day/week calendar code, so the calendar is "Deferred" on both clients, not deleted outright.
- **Stats period toggles (#84, #85):** the PWA added day/week/month period toggles to categories/chores, removed chore date pickers, and scoped indicators/volume to the selected period (sends `?period=day|week|month`). **→ Audited:** this is a *structural* divergence, not a missing toggle. The backend `period=` param is **backward-compatible** (additive; iOS's existing calls still work), so there is **no correctness bug**. iOS `StatsView` implements a *different* design (daily/weekly/monthly time-series charts + baby-care columns; it does not call `/api/stats/breakdown` at all). Converging iOS's stats UX with the PWA's category/chore period model is a **design-led task** that needs on-device/visual verification — deliberately not implemented blind here. Catalogued in the matrix's "Chore stats" row.
- **Log semantics — follow-up / backdate / timezone fixes** (`a8310da`, `e050ec3`, `1a5d5ac`): **→ Audited — iOS already correct, no change needed.** All three were *PWA-specific* bugs caused by the PWA pre-filling a **rounded** schedule hour as `completedAt`, which the shared backend then misclassified as backdated and skipped follow-up clearing. iOS sends `completedAt = Date()` (the real wall-clock time, absolute UTC via `ISO8601DateFormatter`) and a **local** `date` across *every* log path (`LogSheet` default `whenDate = Date()`, `ScheduleView.tapLog`, `QuickLogSheet`), so it never trips the spurious-backdate path. The current backend rule (`log.go:197-202`) treats a log as backdated only when `completedAt < now − 6h`; iOS's "log now" is always within tolerance, so follow-ups clear/replace and `lastFollowUpMinutes` persists correctly.

### 3.3 Endpoint-coverage gaps (static analysis)

Endpoints the **PWA calls but iOS source never references** (candidate missing features):

- `/api/auth/email/verify`, `/api/auth/email/verification/resend` (email verification flow)
- `/api/chores/{id}/restore-default`
- `/api/schedules/for-date` (calendar schedule overlay)
- `/api/stats/breakdown`, `/api/stats/chores/{id}/time-series`, `/api/stats/feeding-gaps`

A few others (`/api/households/{id}/activate`, `/api/notifications/{id}/read`, `/api/notifications/{id}`) are built via path-prefix string concatenation in Swift, so they *may* be covered — verify per endpoint.

### 3.4 The APNs gap — defined contract, no implementation on either side *(confirmed)*

This is a genuine hole, more precisely than my first pass stated:

- **iOS**: `API/RequestModels.swift` defines `APNsRegisterRequest` / `APNsUnregisterRequest` structs (token / environment / bundleId / deviceName) — but a grep confirms they are **referenced nowhere else in the codebase**. There is no `registerForRemoteNotifications`, no `UNUserNotificationCenter`, no `didRegisterForRemoteNotifications`, and no call to a mobile endpoint. The data contract is dead code.
- **Backend**: `/api/mobile/apns/*` is **not routed in `internal/app/server.go`**, and a grep for `apns`/`/api/mobile` across `internal/` + `cmd/` and *every* remote branch returns nothing. The only push routes are the PWA's Web Push (`/api/push/subscribe`, `/api/push/unsubscribe`).

So despite the matrix marking **APNs "Phase 10 — Done,"** native iOS push is **non-functional end to end.** Decide explicitly: implement it (client registration + backend routes + APNs key plumbing) or mark it un-built. Until then, iOS users get no reminders — the app's core value prop. **→ Addressed:** the matrix now marks APNs **Not built**, and `docs/apns-implementation-plan.md` specifies the full client+backend build for when Apple credentials are available.

### 3.5 Why drift is invisible (the enforcement is theater)

There *is* parity tooling — `scripts/check-parity.sh`, `scripts/pre-push-hook.sh`, a `parity` CI job, and an opencode skill — but it doesn't do what it appears to:

- **The CI `parity` job only greps the PR *description*** for one of three magic phrases (`"PWA and iOS both updated"`, `"PWA-only change…"`, `"iOS-only change…"`). It never inspects the iOS diff, never runs the matrix checker, and never builds iOS. Any PR that types the phrase passes — so with the PWA moving ~10× faster, every PWA PR just asserts "PWA-only" and CI stays green.
- **`check-parity.sh` (the real checker) is effectively unrunnable as a gate**: it exits 1 if *any* matrix row is "iOS pending," and ~50 rows are. It can never pass against the current matrix, which is presumably why CI uses the weak description-grep instead.
- **iOS is never compiled or tested in CI.** No macOS runner, no `xcodebuild`. The 11 XCTest suites + UITests exist but are unrun, so an API/model change can break iOS and no automated signal fires.

> **Confirmed, and worse than "unrun": the test target is already red on `main`.** Running `xcodebuild test -only-testing:NabuTests` **fails to compile** (`** TEST FAILED **`): `RequestEncodingTests.swift:82` and `:95` call `CreateChoreRequest(...)` without the `followUpEnabled` parameter that the request model has since gained (the follow-up-scheduling feature). The *app* target compiles, but the *test* target does not — which is only possible if no one has run the iOS tests since that model changed. This is the single most concrete proof of the drift: the iOS test suite is broken on the default branch and nothing noticed.

### 3.6 Recommendations (iOS drift)

1. **Re-baseline `client-parity.md` against reality first.** Walk each row against the code on `main`: most "iOS pending" rows are actually built and should flip to "Done" or "Done (needs re-verify)". This single action restores the matrix as a trustworthy planning tool and is the prerequisite for everything below.
2. **Replace description-grep enforcement with something real.** Once the matrix is accurate, either (a) wire `check-parity.sh` into CI as a blocking gate (now possible because it won't be permanently red), or (b) keep the PR-statement check but *also* require the matrix to be touched when `web/static/js/**` or shared API handlers change. Option (a) is stronger.
3. **Fix the broken `NabuTests` target first** (add `followUpEnabled:` to the two `CreateChoreRequest` calls in `RequestEncodingTests.swift`), then **add an iOS CI lane** (`xcodebuild test` on a `macos-latest` runner) starting with the fast XCTest contract/unit suites so model/API drift fails at PR time. There is no point adding the lane until the suite is green again.
4. **Resolve APNs (§3.4)** explicitly — implement or mark un-built; don't leave it "Done" while non-functional.
5. **Triage §3.2 divergences now** (Activity day/week removal, stats period toggles, log-semantics fixes) before they compound into a large reconciliation later.

## 4. Architecture & Code Quality

**Overall: clean and idiomatic.** Handlers → services → stores layering with dual in-memory/Postgres store implementations is excellent for testability. Stdlib-only HTTP with Go 1.22+ path values; no framework; buildless ES-module frontend with a clever version-injecting cache-buster.

Minor opportunities:

- **`NewServerWithDB` is ~350 lines** of wiring + route registration (`app/server.go:44`). Extract route registration into per-feature `registerXRoutes(mux, …)` functions for readability. Low priority.
- **Reminder scheduler uses `context.Background()`** with no shutdown wiring shown (`server.go:131`). Confirm `cmd/server/main.go` cancels it on SIGTERM for clean shutdown.
- **Tech debt is low** — only 4 TODO/FIXME markers across Go + JS.

## 5. Testing

**Strong on the backend/PWA side:** 45 Go `_test.go` files (race detector + Codecov), 152 JS spec files incl. 42 Playwright E2E specs. Memory stores make service tests fast and DB-free.

Gaps:

- **iOS is never compiled or tested in CI** (§3.5) — the largest testing gap, and a direct driver of invisible drift. 11 XCTest suites + UITests exist but never run; **the `NabuTests` target is currently broken on `main`** (won't compile — confirmed by build).
- **The parity CI job is a description-grep, not a real check** (§3.5). It gives a false sense of safety.
- Consider a **contract test that asserts every PWA-referenced API path has an iOS counterpart** (or an explicit allow-list), to catch §3.3 gaps automatically.

## 6. Performance & Scalability

Fine for the current **single-node** deployment, but two things **block horizontal scaling** and are worth noting before any multi-instance move:

- **The rate limiter is in-process** (`middleware/ratelimit.go`) — per-instance buckets, so N replicas → N× the effective limit. Would need Redis/shared store.
- **The reminder scheduler runs unconditionally on startup** (`server.go:131`) with **no leader election**. Two instances → **duplicate reminder notifications.** Add a single-runner guard (advisory lock / leader election) before scaling out.

Sessions and tokens are already in Postgres (good — no sticky-session requirement there).
